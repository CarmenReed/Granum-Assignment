SECTION E

Common Pushback Questions

Six probes Jonathan is likely to land on the v2 branch. Each one has a head (the probe) and a body (the answer).
"Why did you stage stubs at all?"
Earn-your-complexity at the meta level. Don't implement before validating which axis matters first. The specs carry open questions on purpose, those are the conversation.
"Industry pattern recognition is just plausible-sounding reasoning."
Yes, and it's marked as such in every footnote where it applies. The regulatory citations (Bill 101, 40 CFR Part 170, PCPA, SSA SSN format) are facts. The customer-segment claims are pattern recognition, called out explicitly. Honest framing beats fake citations.
"Did you actually evaluate alternatives?"
No. Each spec is one reasonable shape, not a comparative analysis. If you want a comparative evaluation, that is a different deliverable. This branch shows architectural thinking across five axes, not a procurement decision on each.
"If you could only ship ONE in the next quarter, which one?"
PII expansion. The patterns are written, the patch is build-validated, the test-case file lists the wiring work, and the architecture is unchanged. Highest value per hour of engineering. The reason it is not implemented already is the collision-test risk on each pattern (SSN vs phone fragments, DOB vs locale ambiguity). Those are careful steps, not blocked ones.
"Webhook out has too many open decisions to be useful."
Agree. That is why it is the integration surface, not the integration. HMAC signing, retry/backoff, per-tenant URL config, async delivery channel are real decisions that come before code. The contract is shaped, the wiring point is identified, the call is yours to make.
"Why isn't the language one English/Spanish? That is the bigger workforce reality."
It will be. Operator output language at launch is English or French because LMN's Canadian operator base has a regulatory floor (Bill 101) that cannot be deferred. Spanish is on the crew-input side from day one (any supported language detected). Adding Spanish to the operator-output side is the next jurisdiction expansion, not a launch blocker.
SECTION F

Things NOT to Oversell

Four ways the v2 pitch goes sideways. Pulled from the README's "What NOT to oversell" plus one new one.
These are static contracts and patches, not live wired-up code.
Every stub method throws NotImplementedException (except the compliance detection stub, which returns None deliberately). The PII expansion lives as a .patch file, not as a live source-tree change. None of these enhancements ship runtime behavior on this branch.
I did NOT evaluate alternatives for any of them.
The architectures sketched in the specs are one reasonable shape each, not a comparative analysis. If Jonathan asks for a comparative evaluation, that is a different deliverable.
Customer-pain framing is informed reasoning, not validated customer interviews.
Grounded in public data and industry patterns: Bill 101/96, EPA WPS, Health Canada PCPA, BLS occupational data, SSA SSN format, Granum's stated multi-product strategy. Every spec carries footnotes that mark each claim's epistemic status (regulatory fact / public statistical pattern / industry pattern recognition / unvalidated assumption).
Don't import ghost features.
If something isn't in granum-v2/proposed-code/ or in a .patch file on this branch, it isn't on the branch. Talk about what is staged, not what could be.
